## B1 ‚Äî Drag-Drop legt Knoten an falscher Stelle ab

**Status:** **OFFEN** (tatsaechlich offen, kein Stale-Build-Effekt)

**Steps:**

1. ERD-Tab aktiv (test-schema.erd, 35 Knoten, Zoom 0.1, Pan (354.4, 23.84))
2. Pane-Mittelpunkt berechnet: `(524, 386)` Browser-Pixel (Pane-Rect: left=68, top=85, width=912, height=601.5)
3. CustomEvent `viso-spawn-node` mit `{ type: 'table', clientX: 524, clientY: 386 }` ueber `window` dispatched
4. DOM nach 300ms inspiziert: Knoten `table_36` neu, Style `transform: translate(456px, 300.75px)`, `visibility: hidden`

**Expected (mit `screenToFlowPosition` kanonisch):**

```
flowX = (524 - 68 - 354.4) / 0.1 = 1016
flowY = (386 - 85 - 23.84) / 0.1 = 2769
```

**Actual (buggy `clientPos.x - paneRect.left`):**

```
flowX = 524 - 68 = 456
flowY = 386 - 85 = 300.75
```

**Pixel-Offset im Flow-Space: (-560, -2468) Pixel.** Der Knoten landet weit ausserhalb der gewollten Drop-Position ‚Äî bei Zoom 0.1 fast 2.5k Pixel daneben.

**Frequency:** 1/1 reproduziert (deterministisch)

**Severity:** Hoch ‚Äî Kern-Interaktion

**Console:** Keine Errors, keine Warnings spezifisch zu Drop

**Klassifikation:** `offen` ‚Äî Plan-Hypothese exakt bestaetigt. `clientPos - paneRect.left` ignoriert Viewport-Transform (Zoom + Pan).

**Hub-relevant:** Nein

**Naechster Schritt:** E2.B1 ‚Äî Coordinate-Transform-Unification mit `screenToFlowPosition` aus `useReactFlow()`. Plan-Anhang B detailliert die 4-Schritt-Refactor-Reihenfolge.

## B4 ‚Äî Linien lassen sich nicht loeschen

**Status:** **OFFEN ‚Äî REPRODUZIERT**

**Steps:**

1. Landscape-Tab aktiv, 5 Edges sichtbar
2. Erste Edge anklicken via `pointerdown/mousedown/pointerup/mouseup/click` auf den `react-flow__edge-interaction`-path (interactionWidth=20px stroke confirmed)
3. Nach 200ms: 1 Edge hat Klasse `selected` (`react-flow__edge ... selected selectable`)
4. Backspace-Event auf `document` dispatched (`keydown/keypress/keyup` mit `key='Backspace'`)
5. Nach 500ms: DOM und Source-File geprueft

**Expected:** Edge verschwindet aus Canvas, Source-Datei `landscape.landscape.json` hat eine Relation weniger (4 statt 5)

**Actual:** Edge bleibt im Canvas, Source-Datei unveraendert (relations: 5/5). delta=0.

**Frequency:** 1/1 reproduziert

**Severity:** Hoch ‚Äî Edge ist selektiert (visuelles Feedback funktioniert), aber Loesch-Aktion ohne Effekt

**Console:** Keine Errors

**Klassifikation:** `offen` ‚Äî Edge-Selection greift, aber `onEdgesDelete` wird nicht ausgeloest oder schreibt nicht in Source. Ueberraschend, weil Solution-Doc `react-flow-edges-readonly.md` (Commit `51ce91e`) genau dies gefixt haben soll. Hypothesen:

- (a) Backspace-Event-Dispatching auf `document` statt `window` greift nicht ‚Äî xyflow erwartet Event auf einem anderen Target
- (b) `defaultEdgeOptions.deletable: true` fehlt
- (c) `nodesDelete`/`edgesDelete` Props sind verwendet, aber Synthetic-KeyboardEvent triggert sie nicht
- (d) Realer Bug ‚Äî `onEdgesDelete`-Handler verwirft den Delete

**Bonus-Befund:** `interactionWidth=20px` (xyflow-Default) bestaetigt fuer alle Custom-Edges in Landscape. Plan-Empfehlung 40px steht.

**Hub-relevant:** Nein

**Naechster Schritt:** E2.B4 ‚Äî Real-Test in E4 vor Code-Aenderung (gleiches Diagnose-Limit wie B3). Falls in Real-Test bestaetigt: `defaultEdgeOptions={ deletable: true, interactionWidth: 40 }` plus `onKeyDown`-Audit.
